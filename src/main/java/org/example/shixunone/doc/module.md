# 系统模块设计文档

---


### 1. 系统总体架构设计

本系统采用**前后端分离架构**，基于Spring Boot (后端) + Vue3 (前端) 构建。核心设计原则是**高内聚、低耦合**，并通过中间件（Redis, RabbitMQ）解耦核心业务。

#### 1.1 技术架构栈
*   **前端**：Vue3 (负责页面展示、用户交互、Token管理)
*   **后端**：Spring Boot (核心业务逻辑、接口提供)
*   **数据层**：MySQL (持久化存储)、Redis (会话缓存、Token管理)
*   **中间件**：RabbitMQ (异步解耦、流量削峰)

### 2. 模块详细设计与接口定义

本节详细描述各核心模块的功能点及接口规范。

#### 2.1 用户中心模块 (User Center)
**核心职责**：负责用户身份认证、Token管理及资产模拟。

*   **依赖关系**：
    *   **依赖**：Redis (存储Token), MySQL (用户数据)
    *   **被依赖**：所有其他业务模块（均需鉴权）

*   **功能接口设计**：

| 接口名称 | 请求路径 | 方法 | 请求参数 | 响应数据 | 说明 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| 用户登录 | `/api/user/login` | POST | phone, pwd | `{token, userId}` | 验证通过后生成UUID存入Redis |
| 用户注册 | `/api/user/register` | POST | phone, pwd | `{success}` | 简单注册，无需复杂校验 |
| 获取资产 | `/api/user/balance` | GET | Header: Token | `{balance}` | 需校验Token有效性 |
| 申请提现 | `/api/user/withdraw` | POST | Header: Token, amount | `{msg}` | 仅扣除余额，模拟操作 |

#### 2.2 商品中心模块 (Product Center)
**核心职责**：商品信息的维护与展示。

*   **依赖关系**：
    *   **依赖**：MySQL
    *   **被依赖**：订单模块（下单时读取商品信息）

*   **功能接口设计**：

| 接口名称 | 请求路径 | 方法 | 请求参数 | 响应数据 | 说明 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| 商品列表 | `/api/product/list` | GET | page, size | `List<Product>` | 展示商品名、价格、库存 |
| 发布商品 | `/api/admin/product` | POST | name, price, stock, commission | `{id}` | 管理员接口，需配置固定佣金 |
| 商品详情 | `/api/product/detail` | GET | id | `Product` | 查看具体商品信息 |

#### 2.3 订单与交易模块 (Order & Trade)
**核心职责**：处理模拟购买流程，核心在于**状态流转**与**消息发送**。

*   **依赖关系**：
    *   **依赖**：MySQL (写订单), RabbitMQ (发送消息)
    *   **被依赖**：分销模块（消费者监听）

*   **功能接口设计**：

| 接口名称 | 请求路径 | 方法 | 请求参数 | 响应数据 | 说明 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| 立即购买 | `/api/order/buy` | POST | productId | `{orderId}` | 跳过购物车，生成"待支付"单 |
| 模拟支付 | `/api/order/pay` | POST | orderId | `{success}` | 直接将状态改为"待收货" |
| 确认收货 | `/api/order/confirm` | POST | orderId | `{success}` | **核心节点**：发送MQ消息 |

#### 2.4 分销与返利模块 (Distribution & Rebate)
**核心职责**：关系绑定与异步佣金计算。

*   **依赖关系**：
    *   **依赖**：RabbitMQ (监听队列), MySQL (更新余额)
    *   **被依赖**：无（后台运行）

*   **功能逻辑设计**：
    1.  **关系绑定**：用户注册时解析URL参数 `?invite_id=xxx`，写入数据库 `parent_id` 字段。
    2.  **异步结算 (Listener)**：
        *   监听队列：`commission.queue`
        *   逻辑：收到订单ID -> 查询订单 -> 查询上级ID -> **MySQL事务更新余额** -> 写入返利记录表。

### 3. 核心业务流程依赖分析

根据需求文档中的“异步返利结算流程”，各模块的交互时序如下：

1.  **触发 (订单模块)**：用户点击“确认收货”，订单模块将订单状态更新为“已完成”，并调用 `RabbitTemplate` 发送消息 (`OrderPaidEvent`)。
2.  **传输 (RabbitMQ)**：消息中间件接收消息，暂存于 `commission.queue` 队列中。
3.  **处理 (分销模块)**：后台消费者监听到队列消息，获取订单ID。
4.  **持久化 (数据层)**：分销模块查询MySQL获取订单详情和上级ID，开启事务更新用户余额。
5.  **结束**：事务提交，返利完成。
